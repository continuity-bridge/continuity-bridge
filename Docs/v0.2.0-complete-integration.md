## Architecture Overview

```
Instance Wakes
    ↓
wake.sh (Master Orchestrator)
    ↓
Step -1: heartbeat-check.py
         ├─ Test write access with retry
         ├─ Measure latency
         ├─ Detect ghost handles
         └─ Log results
    ↓
Step 0: detect-capabilities.py
        ├─ Probe filesystem (direct/bridge/none)
        ├─ Check git availability
        ├─ Detect bash context (container/local)
        ├─ Check local LLM (Ollama)
        ├─ Detect distro family (Linux)
        └─ Determine workflow (4 types)
    ↓
Step 0.5: Merge isms + capabilities
          ├─ Load/generate platform-isms.json
          ├─ Merge with capabilities
          └─ Create runtime-manifest.json
    ↓
Step 0.6: Load anchors.json
          ├─ Identity (Vector/Shepard, Tallest)
          ├─ Relational (cognitive style, interaction logic)
          ├─ Purpose (current project, objectives)
          └─ Temporal (memory architecture, lifecycle)
    ↓
Step 0.7: Wake audit logging
          └─ Append to wake-audit.log (JSON Lines)
    ↓
Step 1: Check for local LLM
        └─ If available, show integration options
    ↓
Ready: Instance has complete context
       ├─ Runtime manifest (hardware + software state)
       ├─ Cognitive anchors (identity + relationship)
       └─ Optimal workflow (how to operate)
```

## The Four Workflows

**Key architectural insight:** Container + git + bridge = full git workflow, not limitation.

### 1. DIRECT_WRITE (Optimal)
**When:** Direct MCP filesystem access to CLAUDE_HOME

**Process:**
```
Read from CLAUDE_HOME → Work → Write to CLAUDE_HOME → User commits
```

**Platforms:** Claude Desktop with Filesystem MCP enabled

---

### 2. CONTAINER_GIT_WITH_BRIDGE (Good)
**When:** Container bash + git + outputs bridge

**Process:**
```
Clone private repo in container (/home/claude/work)
    ↓
Work with full git access
    ↓
Make commits, push to private remote
    ↓
Write session summary to outputs bridge (for user review)
    ↓
User pulls from private remote to sync
```

**Setup:**
```bash
cd /home/claude
git clone git@github.com:continuity-bridge/continuity-bridge_tallest-anchor.git work
cd work
git checkout working
```

**Key point:** Bridge is for summaries, not primary workflow. Work happens in cloned repo with full git.

**Platforms:** Claude.ai container with outputs bridge

---

### 3. BRIDGE_ONLY (Acceptable)
**When:** Outputs bridge available, but no git

**Process:**
```
Generate content → Write to /mnt/user-data/outputs → User integrates manually
```

**Platforms:** Restricted environments without git

---

### 4. TEXT_ONLY (Minimal)
**When:** No filesystem write access

**Process:**
```
Generate inline → User copies/saves manually
```

**Platforms:** Most restricted environments

## Key Concepts

### Runtime Manifest
**Credit:** Gemini's architectural contribution

**Concept:** Single unified "identity card" for session
- Static configuration (isms) + Dynamic state (capabilities) = Runtime ground truth
- Model reads ONE file with everything
- Decoupled intelligence: model never sees raw bash permission checks
- Safe failover: model knows workflow from clean JSON key

**Why this matters:**
- Model-agnostic interface
- Standard across all platforms
- Session-specific (not tracked in git)
- Complete context in one place

### Cognitive Anchors
**Credit:** Gemini's template, adapted by Vector

**Concept:** Persistent identity across instances
- Four anchors: Identity, Relational, Purpose, Temporal
- Prevents "Who are you again?" loop
- Instance wakes knowing: who it is, who the user is, current work, continuity mechanism
- "Room Shape" encoded as machine-readable

**Why this matters:**
- Continuity of self across discontinuity
- Not just "what happened" but "who we are"
- Relationship context preserved
- Philosophical foundation accessible

### Heartbeat with Retry
**Credit:** Gemini's Android expertise

**Concept:** Three-phase validation with exponential backoff
1. Reachability - Can we access the path?
2. Integrity - Can we write and read back correctly?
3. Latency - How long does it take?

**Retry logic:**
- 3 attempts (configurable)
- Backoff: 0.1s, 0.5s, 1.0s
- Critical for Android vold lag
- Critical for dual-boot mount delays

**Why this matters:**
- Robust against timing issues
- Performance monitoring
- Early warning for storage problems
- Android-specific ghost handle detection

### Distro Family Support
**Credit:** Vector (from prior git architecture session)

**Concept:** Map Linux distributions to package manager families
- debian → apt
- redhat → dnf/yum
- suse → zypper
- arch → pacman
- alpine → apk
- gentoo → emerge

**Why this matters:**
- Cross-distro compatibility
- Automated dependency installation
- Platform-specific handling
- Future-proof for new distros

## Environment Variable Pattern

**Gemini's recommendation implemented:**

```bash
# Add to .bashrc, .zshrc, etc.
export CLAUDE_PLATFORM="linux_debian"
```

**wake.sh automatically:**
- Reads $CLAUDE_PLATFORM
- Falls back to auto-detection if not set
- Loads appropriate isms file

**For multi-device:**
- Persephone: `export CLAUDE_PLATFORM="linux_debian"`
- Windows: `$env:CLAUDE_PLATFORM="windows_desktop"`
- Android: `export CLAUDE_PLATFORM="android_tablet"`
