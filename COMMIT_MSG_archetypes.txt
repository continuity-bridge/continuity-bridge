## How It Works

### 1. User Selects Archetype (Onboarding)
```bash
cp anchor-templates/creative-anchor.json anchors.json
# Then customize with their details
```

### 2. System Detects Tools Based on Archetype
```python
# Creative archetype → checks Blender, GPU, Adobe
# Social archetype → checks API access, scheduling tools
# Executive archetype → checks project mgmt, BI tools
# etc.
```

### 3. Runtime Manifest Includes Archetype Context
```json
{
  "archetype": "creative",
  "tools": {
    "archetype_specific": {
      "blender": true,
      "nvidia_gpu": true
    }
  }
}
```

### 4. Instance Behavior Adapts
- Creative: Preserves aesthetic notes, tracks style guides
- Social: Monitors engagement, maintains brand voice
- Executive: Identifies blockers, tracks team bandwidth
- Pedagogical: Tests understanding, connects concepts
- Wellness: Tracks energy patterns, respects capacity

## Key Concepts

**Functional Filters** (Gemini's insight):
Archetypes aren't just documentation - they actively filter what the
system checks for and how it behaves.

**Horizontal Scalability**:
Same Continuity Bridge engine serves:
- Developers (technical)
- Artists (creative)
- Influencers (social)
- Managers (executive)
- Students (pedagogical)
- Health-focused users (wellness)

**Archetype Blending**:
Users can combine archetypes:
- Technical + Wellness (managing ADHD while coding)
- Creative + Social (artists building audience)
- Executive + Pedagogical (learning while leading)

## Integration with v0.2.0

Seamlessly integrates with existing wake system:

```
wake.sh

  ├─ Reads archetype from anchors.json
  ├─ Checks archetype-specific tools
  └─ Adds to runtime-manifest.json
  ↓
Instance wakes with archetype-aware context
```
